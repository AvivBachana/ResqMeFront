🎯 מטרת העל

להעניק למתכנת תמונה מלאה, קצרה ומעשית של רכיבי הבקאנד הקיימים, מה מותר ומה אסור לשנות, איך להריץ בדיקות, ואיך להמשיך לפתח בעזרת GitHub Copilot בלי לגרום לו לשכתב את הפרויקט או להוסיף רכיבים לא רלוונטיים.

⚠️ עמדה מקצועית וגבולות פיתוח

כרגע יש להתמקד **באינטגרציה בלבד**! אין להוסיף רכיבי TTS, אין להוסיף יצירת דאטה, אין לשנות את מנוע ה-DSS ואין לבנות מחדש ארכיטקטורה. כל שינוי כזה יגדיל סיכונים ויבזבז זמן עבודה יקר.

1 תקציר המצב הנוכחי

ישנה חבילת backend פשוטה ושטוחה בשם resqme_backend_simple. החבילה מיועדת להיות נקודת האמת הבלעדית של הבקאנד לצורך חיבור הפרונט. היא כוללת FastAPI, מנוע DSS, קובץ Excel שהוא בסיס הידע, רכיב STT בעברית, וקבצי בדיקה מינימליים.

  • הפרונט לא אמור לקרוא את קובץ ה-Excel dss.xlsx ישירות.
  • הפרונט שולח טקסט או קובץ אודיו ל-API ייעודי.
  • ה-Backend מפעיל את מנוע ה-DSS ומחזיר JSON מובנה ומוכן להצגה ישירה.
  • רכיב ה-STT הוא רק שלב תמלול (המרת קול לטקסט), הוא אינו מחליף את מנוע ה-DSS.
  • קובץ ה-Excel מהווה את בסיס הידע של המערכת והוא חובה מוחלטת להרצת המערכת.

3 תפקיד כל קובץ

שם הקובץ תפקיד במערכת חובה?
main.py שרת FastAPI. מגדיר את ה-endpoints שהפרונט צריך לקרוא. כן
schemas.py מודלים של Pydantic המגדירים את מבנה הבקשה והתשובה. זה חוזה ה-API. כן
dss_adapter.py שכבת תיווך בין ה־API לבין מנוע ה־DSS. מקבל טקסט ומחזיר פלט אחיד. כן
stt_adapter.py שכבת תיווך בין קובץ אודיו לבין STT. מחזיר תמלול נקי שמועבר ל־DSS. רק לאודיו
dss_engine.py מנוע ההחלטה. מכיל את הלוגיקה שמסווגת אירוע רפואי לפי תסמינים. כן
excel_dss_loader.py טוען את dss.xlsx וממיר אותו למבני נתונים שהמנוע יודע לעבוד איתם. כן
dss.xlsx בסיס הידע של ה־DSS: מצבים, מילות מפתח, חוקים והנחיות. קריטי

4 מהו dss.xlsx ומה הוא לא?

קובץ ה-Excel הוא בסיס הידע (Knowledge Base). בלעדיו המנוע לא מכיר שום חוק או מצב רפואי. החלוקה היא ברורה:

קובץ המידע dss.xlsx Knowledge Base
⬇️
טעינה excel_dss_loader.py Loads Knowledge Base
⬇️
סיווג dss_engine.py Decision Engine
⬇️
חוזה API main.py / dss_adapter.py Exposes Logic

5 התקנה והרצה מקומית

יש לפתוח את התיקייה כפרויקט עצמאי ולהריץ:

python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn main:app --reload

בדיקות חובה (הרצה מקומית לפני חיבור הפרונט!)

python test_text_flow.py
python test_dss.py

* ציפייה: test_dss.py אמור להחזיר Passed: 15/15.

7 הנחיות עבודה נכונות עם GitHub Copilot

🚫 דברים שאסור לתת לקופיילוט לעשות!

  • לא לשכתב את dss_engine.py ללא סיבה קונקרטית מוגדרת.
  • לא למחוק או לשנות את קובץ האקסל dss.xlsx.
  • לא לאפשר לפרונט לקרוא או לעבד את האקסל ישירות.
  • לא להוסיף רכיבי TTS (Text-to-Speech) או רכיבי יצירת דאטה דמו.
  • לא לשלב בסיסי נתונים חדשים (כמו Firebase) אם לא נתבקשת במפורש.
💬 פרומפט פתיחה לקופיילוט

You are working on an existing ResQme frontend and backend integration. The backend folder is the source of truth. Do not restructure the backend. Do not rewrite dss_engine.py. Do not modify dss.xlsx unless explicitly asked. Do not add TTS or data generation. Do not hardcode medical results in the frontend. First, inspect main.py, schemas.py, dss_adapter.py, and README.md. Then explain the exact API calls the frontend should use before changing code.

💬 פרומפט לחיבור קלט טקסט

Connect the existing text input in the frontend to the backend endpoint POST /analyze-text-text. Send JSON with the exact shape: { "text": <typed user text> }. After receiving the response, store these fields in frontend state: case_id, condition, urgency, confidence, transcript, instructions, warnings, matched_symptoms, alternative_conditions. Then navigate to the result screen and bind these fields to the existing UI components. Do not change backend logic. Do not hardcode any DSS result in the frontend.
